Program Setup — Custom PaaS Application
Enterprise Client · Deloitte
The standard Oracle Program page offers limited field coverage out of the box and provides no mechanism for maintaining audit history — two requirements that were non-negotiable for the enterprise client. Rather than working around the native page's constraints, the decision was made to architect a fully custom PaaS application from the ground up. Program Setup Information was designed and built on Oracle Visual Builder Cloud Service, replacing the standard Oracle Program page entirely and extending its capabilities to meet the business's real-world complexity. The application is structured around three distinct program types — Parent, Child, and Standalone — each with its own field configuration and validation rules. Hierarchy relationships are enforced at the data level, with Parent-to-Child linkages maintained in Oracle ATP and validated on every create and update operation to ensure the program structure remains consistent and navigable. The home page is built around an advanced filter panel, enabling programme managers and administrators to search across the full program estate using a combination of status, type, date range, responsible person, and programme-specific attributes — giving users the search precision that the standard Oracle page's basic lookup could not provide. The application exposes full View, Create, and Update workflows at the program level. The create and update forms span 40+ fields, covering programme metadata, scheduling information, financial associations, and people-related data. People fields — including Programme Manager, Senior Responsible Officer, and key stakeholder roles — were implemented using complex reusable VBCS fragments, encapsulating person search, selection, and display logic in self-contained components that are reusable across forms and maintainable independently of the parent page. Data sourcing was carefully architected to leverage Oracle's existing master data rather than duplicating it. Standard Oracle REST APIs are used to pull lookups, value sets, and reference data — including programme categories, status codes, and organisational units — ensuring the PaaS application remains aligned with the ERP system of record without maintaining parallel data copies. Core programme records are persisted in Oracle ATP, with PL/SQL procedures handling the validation and storage logic to keep business rules centralised and consistent. The most technically demanding aspect of the application is the cross-field validation layer. Across 40+ fields, validation rules govern permitted value combinations, mandatory field conditions based on programme type, date sequencing constraints, and lookup value restrictions. These validations were implemented at both the UI layer — providing immediate inline feedback to users — and at the database layer, ensuring data integrity is maintained even if programmatic interfaces bypass the UI. Audit history was implemented as a first-class capability: every field-level change is captured with a before-and-after value, the identity of the user, and a precise timestamp — giving compliance teams a complete, navigable history of every programme record change from the point of go-live forward.

Job Assignment Suite
Enterprise Client · Deloitte
Multi-Level Approvals BI Publisher Reports

The Job Assignment Suite was developed as a comprehensive Oracle VBCS application to manage the end-to-end lifecycle of job assignments across a large enterprise workforce. Prior to this solution, the client relied on manual, spreadsheet-driven processes to track and manage thousands of job assignments annually — a workflow riddled with data errors, approval bottlenecks, and zero auditability. The application was built on Oracle Visual Builder Cloud Service, leveraging the Redwood design system to deliver a modern, intuitive user experience aligned with Oracle's latest design language. At its core, the application manages the full assignment lifecycle: initiation, review, approval, and closure. The home page features a rich search interface with advanced filters, enabling operations teams to quickly surface assignments by status, type, worker, or department. Each assignment record captures a comprehensive set of fields — assignment type, effective dates, reporting structure, compensation details, and associated role information — ensuring nothing falls through the cracks of an otherwise fragmented manual process. Integration was a critical component of the architecture. The application connects to Oracle HCM Cloud via standard REST APIs to pull worker and position data in real time, ensuring assignment records are always aligned with the authoritative HR system of record. Oracle Integration Cloud (OIC) was used to orchestrate multi-step integration flows, handling data transformation, error management, and retry logic to ensure reliable, auditable data movement between VBCS and the backend HCM platform. For data persistence, Oracle Autonomous Transaction Processing (ATP) was leveraged as the backend database, storing assignment records, approval audit trails, and workflow state. PL/SQL procedures were written to handle complex business logic — assignment validation rules, duplicate detection, and historical record management. BI Publisher was integrated to generate automated assignment reports, giving managers on-demand access to formatted, printable summaries in PDF and Excel formats for compliance and HR audit purposes. Reports could be triggered directly from within the application. The approval workflow was designed as a configurable, multi-level process, routing assignment records through HR Business Partners and line managers based on assignment type and organisational hierarchy. Each approval action was timestamped and logged, building a complete audit trail that was previously impossible to maintain in the spreadsheet-based process. The application made extensive use of reusable VBCS fragments for worker search, position lookup, and date-range selectors — patterns that were subsequently reused across other applications delivered by the team. Field-level validations enforced business rules inline, catching errors before they could propagate downstream into HCM.

The framework was architected from the ground up to provide reusable, maintainable, and scalable test automation across Oracle Fusion modules. Journal Entry Checklist Application
Enterprise Client · Deloitte
Approval Workflow Audit Compliance
Read full description
The Journal Entry Checklist Application was built to introduce a structured, enforced governance layer into the financial close process for a large enterprise running Oracle Fusion Finance. Prior to this solution, Journal Entries were being prepared and loaded into the Oracle General Ledger without a formalised pre-submission review mechanism. This created risk: entries could be uploaded with errors, missing supporting documentation, or without the sign-off of the appropriate financial controller — all of which introduced audit exposure and downstream reconciliation effort. The application was developed on Oracle Visual Builder Cloud Service, integrating directly with Oracle Fusion Finance via standard REST APIs. Its primary function is to enforce a configurable checklist of pre-submission tasks that must be completed and approved before a Journal Entry batch is permitted to progress to the loading stage. The workflow is initiated when a JE preparer creates a new checklist record linked to a specific JE batch. The preparer works through each required checklist item — attaching supporting documentation, confirming data accuracy, verifying period alignment, and confirming that the appropriate cost centres and accounts are correctly mapped. Each item is individually checked off with a timestamp, creating a granular audit trail that persists independently of the JE record itself. Once all preparer checklist items are completed, the record is routed through a multi-level approval process. The first approval tier is the Finance Team Lead, who reviews the checklist completion evidence and the associated JE data before approving the record to proceed. For JEs above a defined materiality threshold, a secondary approval from the Financial Controller is required, providing an additional layer of governance for high-value transactions. Only after all required approvals have been captured does the application permit the JE batch to be submitted for loading into Oracle ERP — ensuring no entry bypasses the review process. Approvers can reject records back to the preparer at any stage with mandatory comments. All checklist records, approval decisions, attachments, and audit metadata are stored in Oracle ATP, providing a persistent, queryable audit repository. REST API integrations with Oracle Fusion Finance allow the application to pull JE batch details directly into the checklist record, reducing manual data entry and the risk of mismatched information between the checklist and the actual JE in the system.

Read full description
Business Intelligence reporting formed a critical output layer across the Oracle Cloud programme, providing Finance, Procurement, and HR teams with formatted, scheduled, and on-demand reports directly from Oracle Fusion data. The BI Publisher reporting suite was designed and built to replace a mix of manual extracts, spreadsheet-based consolidations, and ad hoc query outputs that could not meet the volume, consistency, or formatting requirements of enterprise stakeholders. BI Publisher was selected as the primary reporting engine due to its deep, native integration with Oracle Fusion data models. Reports were built on top of Oracle's delivered data models where available, and on custom SQL and PL/SQL-based data sets where the standard data models did not surface the required fields or joins. This dual approach allowed the reporting layer to cover both standard Oracle transactional flows and the bespoke business logic introduced through custom PaaS extensions. Report templates were designed in RTF and XLS formats, with careful attention to enterprise formatting standards — including conditional formatting, column grouping, and multi-level summarisation where required. Each report was built to support multiple output formats — PDF for formal distribution and sign-off, Excel for operational use and downstream reconciliation, and HTML for browser-based consumption within Oracle Fusion. Across the Finance workstream, BI Publisher reports covered journal entry listings, account balance summaries, period-close checklists, and intercompany transaction reconciliations. Procurement reports addressed purchase order status, open requisition aging, supplier spend analysis, and receipt-to-invoice matching exceptions. HR reports provided headcount summaries, job assignment listings, and position vacancy reports — all formatted to meet business requirements that standard OTBI output could not satisfy. Scheduling and delivery was managed through Oracle Fusion's report scheduler, with reports configured to run at predefined intervals — daily, weekly, or period-end — and distributed to stakeholder distribution lists via email or placed in designated BI Catalog folders for self-service access. Burst scheduling was used for reports requiring recipient-specific data slices, delivering personalised output to each cost centre owner or department manager without manual intervention. All report templates and data models were maintained with version control discipline, and a naming convention was enforced across the BI Catalog to ensure discoverability. Report performance was regularly tuned through query optimisation and the use of bind variables to prevent full-table scans on high-volume transactional tables.

Oracle Integration Cloud served as the backbone of the integration architecture across the Oracle Cloud programme, connecting VBCS front-end applications, Oracle Fusion ERP modules, external data sources, and downstream reporting systems through a managed, monitored, and governed integration layer. Unix Wrapper Script — BI Report Trigger
Enterprise Client · Deloitte
SOAP API Automation Pipeline Integrated
Read full description
The Custom Unix Wrapper Script Solution was an innovative, pragmatic engineering response to a specific operational challenge: the need to trigger Oracle BI Publisher reports programmatically from outside the Oracle Fusion application context, without relying on Oracle's standard scheduling UI or manual report submission. The solution was designed to support automated, event-driven report generation as part of larger data processing pipelines running on the Oracle Cloud Infrastructure Linux environment. Oracle BI Publisher exposes a SOAP-based Web Services API that allows reports to be submitted, parameterised, and retrieved programmatically. The wrapper script solution was built around this API, encapsulating the complexity of SOAP request construction, authentication header management, parameter injection, and response parsing within a set of reusable Unix shell scripts. This allowed other pipeline components — database jobs, OIC flows, and scheduled tasks — to trigger BI Publisher report execution through a simple, standardised shell invocation without needing to embed SOAP logic directly in each calling process. The core wrapper script accepted report path, output format, and parameter values as command-line arguments, constructing the appropriate SOAP payload dynamically using heredoc templating. Authentication was handled via Oracle Fusion's token-based security model, with credentials managed through encrypted credential files to avoid hardcoding sensitive values in the script logic. The script handled the full SOAP request lifecycle — submission, status polling, and output retrieval — with built-in retry logic for transient network failures and timeout handling for long-running report executions. Output handling was a key capability of the wrapper. Reports could be directed to the local filesystem for subsequent processing by downstream scripts, pushed to Oracle ATP via SQL*Loader for archival and audit purposes, or transmitted to designated email distribution lists through the script's integrated notification layer. This flexibility made the wrapper applicable across a wide range of report delivery scenarios without requiring a new script for each use case. Logging was embedded at every stage of the wrapper's execution lifecycle — submission timestamp, parameter values used, execution duration, HTTP response codes, and output file paths were all written to a structured log file consumed by the operations team's monitoring tooling. The solution was integrated into the period-close pipeline as the final step, automatically triggering formatted report generation immediately after Oracle Fusion processing jobs completed — ensuring finance stakeholders received their reports within minutes of the underlying data being available. The wrapper was designed to be configuration-driven, with report paths, parameter definitions, and delivery targets maintained in external configuration files, so new reports could be added to the automated pipeline by updating configuration alone.
